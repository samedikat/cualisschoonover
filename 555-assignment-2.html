Heuristic Analysis

In my review of the JSTOR interface for the Catholic University of America’s student newspaper archive, I used Nielsen’s ten usability heuristics as a framework. Overall, the system performs well, especially in terms of speed and clarity, but several areas could benefit from refinement, especially when considering undergraduate users or instructors looking to guide students through archival research.

  1. Visibility of System Status
    2. The system responds quickly, and while items (such as scanned PDFs) take a little longer to load, there is a clear animation indicating progress. Breadcrumbs at the top of the page help users understand where they are. These visual cues help users stay oriented—important for students new to archival interfaces. No specific recommendations.

  2. Match Between System and the Real World
    3. Most of the language used is accessible, but one term—“Serial”—could confuse users unfamiliar with library terminology. A hover-over explanation of what “Serial” means in this context would clarify things for undergraduate users.

  3. User Control and Freedom
    4. A “Back to results” link provides easy navigation. However, the collection search doesn't search the full text of the newspapers, which limits its usefulness for research. Adding an advanced search feature and more filtering options would improve usability—especially for students trying to locate specific information within the issues.

  4. Consistency and Standards
    5. The presence of two search bars (one for JSTOR as a whole, one for the collection) is potentially confusing. The JSTOR-wide search remains visible even while viewing items within the CUA collection, often leading to accidental exits. Hiding or disabling the broader JSTOR search while viewing collection items could reduce this confusion.

  5. Error Prevention
    6. The design is simple, and there are few opportunities for user error. One built-in protection is a pop-up dialog asking users to accept terms before downloading files. No recommendations at this time.

  6. Recognition Rather Than Recall
    7. Labels and buttons are clear. Hover text in the item viewer is helpful. No recommendations.

  7. Flexibility and Efficiency of Use
    8. Users can sign in to JSTOR and save materials to a personal workspace. However, more advanced filtering or sorting options would support researchers and instructors alike. Again, more facets and full-text search would help advanced users and make this tool more robust for classroom assignments.

  8. Aesthetic and Minimalist Design
    9. The interface is clean and mostly unobtrusive. Some large thumbnail images at the collection level push actual content below the visible part of the screen. Highlighted search terms in the PDF viewer are small and easy to miss. Resizing images or moving them lower could help. Additionally, offering a clearer summary of search hits within a document might aid navigation, especially for students skimming for relevance.

  9. Help Users Recognize, Diagnose, and Recover from Errors
    10. There aren’t many error messages. The main mistake is using the wrong search bar, which leads to leaving the collection unintentionally. A gentle warning or label could remind users which search bar they are using.

  10. Help and Documentation
    11. JSTOR includes a “Search Help” link, but it gives only general advice. Occasionally, a chat box offers JSTOR support, but it doesn’t seem specific to this collection. It would be helpful to include contact info or a link back to the university’s Libraries or Special Collections—particularly useful for instructors creating assignments or students who have collection-specific questions.

    Summary

    Strengths

    1. Document viewer is clean and easy to use
      2. Once I selected an issue, the PDF viewer opened right in the browser. I didn’t have to download anything, and page navigation was straightforward. This makes it easy for students to scan primary sources without getting bogged down by extra steps.

    2. Simple, unobtrusive design
      3. The site’s layout is straightforward, and it’s clear that the content is the focus. There’s not a lot of visual clutter or unnecessary design elements, which helps reduce distraction. While it might come across as plain, the simplicity serves the archive well.

    Weaknesses

    1. Limited indexing restricts both search and exploration
      2. The site’s search tool appears to only scan item-level metadata, not the full text of the digitized issues. This makes it hard for users to find specific content, particularly since there are also no thematic tags, subject headings, or filters to guide browsing. Together, these omissions make the collection feel flat and hard to navigate, especially for users approaching it without knowing exactly what they're looking for.

    2. Dual search bars create confusion
      3. The site has two distinct search fields—one for the overall JSTOR platform and one for the specific collection. Their visual similarity and proximity could easily lead users to enter terms into the wrong box, especially since only one of them searches within the Catholic University archive. When an item is open, only the JSTOR search bar is visible, which can mislead users who want to perform a new search of the collection after viewing an item. This design choice adds friction and undermines the efficiency of an otherwise straightforward user task.

    Comparison of Results

    Both my heuristic evaluation and usability testing pointed to similar strengths and weaknesses in The Tower archives. My user and I both felt that:

    • The limited search functionality is a major weakness.
      • As I observed in my own testing, user was frustrated by the inability to search full text or filter results. They often struggled to find what they were looking for, reinforcing my conclusion that the system lacks sufficient indexing and browsing tools.

    • The design feels simple and consistent, but not particularly engaging.
      • Although the layout is clean and unobtrusive—something I noted as a strength—my user was lukewarm about its appeal. They selected “neither agree nor disagree” when asked if it invited exploration, suggesting it felt more serviceable than inspiring.

    One small difference stood out:

    • User appreciated visual previews in search results.
      • I didn’t note this in my initial evaluation, but my user specifically mentioned liking the thumbnail images of each issue. These previews gave them a sense of the content before clicking and helped make the results feel a bit more dynamic. This points to the potential value of using visuals more deliberately throughout the site.

    Suggestions for Improvement

    Some of these ideas emerged during my heuristic evaluation, and the usability study reinforced their importance:

    • Improve indexing and search functionality.
      • Full-text search would significantly improve user experience, as would implementing metadata tags or categories to help users browse by topic or theme.

    • Clarify and streamline the search interface.
      • Reducing visual confusion between the JSTOR-wide and collection-specific search bars would prevent user error and improve efficiency.

    • Leverage visuals more consistently.
      • Since users liked the thumbnail previews, consider adding more visual cues—perhaps using highlighted snippets or featured content blocks—to encourage exploration and discovery.
